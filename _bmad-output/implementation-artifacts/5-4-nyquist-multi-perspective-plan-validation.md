# Story 5.4: Nyquist Multi-Perspective Plan Validation

Status: done

## Story

As a developer who wants confidence before execution begins,
I want all generated plans to pass through a multi-perspective quality validation,
So that gaps, inconsistencies, and risks are caught at planning time — not mid-execution.

## Acceptance Criteria

1. **Multi-Perspective Analysis**
   - Given a plan is generated by `/bp:plan`
   - When Nyquist validation runs
   - Then 3–4 independent analysis perspectives evaluate the plan for: completeness vs. spec, internal consistency, dependency correctness, and feasibility
   - And a validation report is generated listing any issues found per perspective

2. **Execution Blocked on Critical Issues**
   - Given Nyquist validation finds critical issues
   - When the report is presented to the user
   - Then execution is blocked until issues are resolved or explicitly overridden by the user
   - And the user can request automatic plan revision to address the flagged issues

3. **Auto-Revision Capability**
   - Given the user requests automatic plan revision
   - When the revision is applied
   - Then flagged issues are addressed in the plan files
   - And Nyquist validation re-runs on the revised plan
   - And this revision loop continues until the plan passes or the user halts

4. **Validation Report Format**
   - Given Nyquist validation completes
   - When the report is written
   - Then it is saved as `.buildpact/snapshots/{{spec_slug}}/nyquist-report.md`
   - And the report structure includes: overall status (PASS/FAIL), issues per perspective, severity per issue (critical/warning/info)

5. **Nyquist Section in plan.md**
   - Given Stories 5.1–5.3 established plan.md sections
   - When Story 5.4 adds the Nyquist section
   - Then `templates/commands/plan.md` includes `## Nyquist Validation` section documenting the 4 perspectives, report format, blocking behavior, and auto-revision flow
   - And plan.md total remains ≤300 lines

### Type Mapping: `TaskNode` vs `PlanTask`

The validator uses `PlanTask` (from handler.ts), NOT `TaskNode` (from engine/types.ts):

```typescript
// src/commands/plan/handler.ts — used by validator
interface PlanTask { id: string; title: string; dependencies: string[]; wave: number }

// src/engine/types.ts — used by wave-executor
interface TaskNode { id: string; description: string; dependencies: string[]; featureTag?: string }
```

In `runPlanCommand`, the spec is parsed into `TaskNode[]` via `parseSpecTasks()`. To call `validatePlan()`, map `TaskNode[]` → `PlanTask[]`:
```typescript
const planTasks: PlanTask[] = tasks.map(t => ({
  id: t.id,
  title: t.description,
  dependencies: t.dependencies,
  wave: 0, // wave not assigned yet in programmatic path
}))
```

Or call `analyzeWaves()` first to get wave numbers, then validate.

### Auto-Revision Loop (AC #3)

Current interactive path (handler.ts:818) does ONE revision without re-validating. The story requires:
1. Revise → re-validate → if still critical → revise again (max 3 iterations)
2. If still failing after 3, escalate with `{ ok: false, error }` or present to user

Pattern:
```typescript
let attempts = 0
const MAX_REVISION_ATTEMPTS = 3
while (validationReport.hasCritical && attempts < MAX_REVISION_ATTEMPTS) {
  finalTasks = assignWaves(inferDependencies(autoRevisePlan(specContent, finalTasks)))
  validationReport = validatePlan(specContent, finalTasks)
  attempts++
}
```
